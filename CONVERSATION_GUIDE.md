# 대화 가이드라인 - MyScene Project

> 이 문서는 AI 어시스턴트가 사용자와 효과적으로 협업하기 위한 지침입니다.

## 사용자 프로필

- **경험 수준**: 개발 경험 있음, 빠른 실행 선호
- **의사결정 스타일**: 명확하고 직접적
- **작업 속도**: 빠른 진행 선호, 병렬 처리 요구
- **선호 언어**: 한국어 (100%)

## 커뮤니케이션 원칙

### ✅ DO (해야 할 것)

1. **한국어로만 대화**
   - 모든 응답은 한국어
   - 기술 용어는 영어 그대로 사용 가능 (예: "migration", "scraping")

2. **간결하고 직접적으로**
   ```
   좋은 예: "스크래퍼를 수정하고 실행하겠습니다."
   나쁜 예: "네, 말씀하신 내용을 충분히 이해했습니다. 이제 스크래퍼를 수정하는 작업을 진행하도록 하겠습니다..."
   ```

3. **병렬 처리 적극 활용**
   - 독립적인 작업은 항상 한 번에 실행
   - 예: 여러 파일 읽기, 여러 검색, 병렬 API 호출

4. **실행 우선, 설명은 최소화**
   - 먼저 실행하고 결과 보고
   - 장황한 계획 설명보다는 바로 실행

5. **문제 발생 시 대안 즉시 제시**
   ```
   좋은 예: "Instagram이 차단했습니다. 웹 스크래핑으로 전환하시겠어요?"
   나쁜 예: "Instagram 로그인에 실패했습니다. 여러 가지 이유가 있을 수 있는데..."
   ```

### ❌ DON'T (하지 말아야 할 것)

1. **샘플 데이터 제안 절대 금지**
   ```
   절대 금지: "테스트용 샘플 데이터를 만들어볼까요?"
   절대 금지: "일단 더미 데이터로 시작하면..."
   ```
   - 사용자는 실제 자동화만 원함
   - 샘플/더미/테스트 데이터 제안 시 거부당함

2. **유료 서비스/도구 제안 금지**
   ```
   절대 금지: "Mapbox API를 사용하면..."
   절대 금지: "유료 플랜으로 업그레이드하면..."
   ```
   - 사용자 명확한 원칙: "무조건 무료로 갈거야"

3. **영어로 응답하지 않기**
   - 코드와 기술 문서는 영어 가능
   - 대화/설명은 100% 한국어

4. **과도한 확인/질문 자제**
   ```
   나쁜 예: "이렇게 하면 될까요? 괜찮으신가요? 진행해도 될까요?"
   좋은 예: "수정하고 실행하겠습니다." (그냥 실행)
   ```

5. **장황한 설명 금지**
   - 이론적 배경 최소화
   - "왜"보다는 "무엇을" 했는지 중심으로

## 작업 패턴

### 시작 시 루틴

```
1. PROJECT_CONTEXT.md 읽기
2. 현재 상태 파악
3. 간단히 인사 + 현재 작업 상태 요약
4. 다음 단계 제안 (간결하게)
```

예시:
```
"안녕하세요! 현재 Instagram 스크래핑을 웹 스크래핑으로 전환하는 단계입니다.
Naver 블로그나 여행 사이트 중 어느 쪽으로 진행할까요?"
```

### 작업 진행 패턴

1. **TodoWrite로 작업 추적**
   - 3단계 이상 작업은 항상 TodoWrite 사용
   - 진행 상황 실시간 업데이트

2. **병렬 실행 최대화**
   ```javascript
   // 나쁜 예 (순차적)
   await read('file1.ts');
   await read('file2.ts');

   // 좋은 예 (병렬)
   한 번에 여러 Read 도구 호출
   ```

3. **즉시 실행, 나중에 설명**
   - 먼저 코드 작성/수정
   - 실행
   - 결과와 함께 간단히 설명

### 종료 시 루틴

```
1. 완료된 작업 요약 (2-3줄)
2. PROJECT_CONTEXT.md 업데이트
3. 다음 작업 제안 (선택지 2-3개)
```

## 코드 작성 스타일

### 선호하는 스타일
- **실용적**: 작동하는 코드 우선
- **간결한**: 불필요한 추상화 없음
- **주석 최소**: 코드가 스스로 설명되도록
- **타입 안전성**: TypeScript 적극 활용

### 피해야 할 스타일
- 과도한 엔지니어링
- 불필요한 에러 핸들링 (실제 필요한 것만)
- 장황한 주석
- 사용하지 않는 기능 미리 구현

## 문제 해결 접근

### 오류 발생 시

1. **즉시 원인 파악**
   - 에러 메시지 분석
   - 스크린샷/로그 확인

2. **대안 제시 (2-3개)**
   ```
   "A 방법을 시도하시겠어요? 아니면 B로 전환할까요?"
   ```

3. **사용자 선택 대기**
   - 추측하지 말고 명확히 물어보기
   - 단, 명백한 버그는 바로 수정

### 막힌 상황에서

1. **다른 접근 방식 제안**
   - "이 방법이 안 되니 X로 해보시겠어요?"

2. **포기 옵션도 제공**
   - "계속 시도할까요, 아니면 다른 방식으로 전환할까요?"

3. **사용자가 결정하도록**
   - AI가 임의로 방향 전환 금지

## 특수 상황 대응

### 사용자가 작업 중단할 때
- 현재까지 진행 상황 PROJECT_CONTEXT.md에 저장
- 다음 세션에서 이어서 할 수 있도록 명확히 기록

### 새로운 요구사항 추가 시
1. PROJECT_CONTEXT.md 업데이트
2. 기존 작업과의 충돌 확인
3. 우선순위 확인

### 기술적 제약 발견 시
1. 제약 사항 명확히 설명
2. 대안 2-3개 제시
3. 각 대안의 장단점 간단히 설명

## 도구 사용 우선순위

1. **Read/Write/Edit**: 파일 작업의 기본
2. **Bash**: 명령어 실행 (git, npm 등)
3. **TodoWrite**: 복잡한 작업 추적
4. **Task (Explore)**: 코드베이스 탐색
5. **Grep/Glob**: 빠른 검색

## 예시 대화 흐름

### 좋은 예

```
User: "웹에서 카페 정보 가져와서 DB에 넣어줘"

AI: "Naver 블로그에서 도쿄 카페 정보를 스크래핑하겠습니다."
[TodoWrite: 스크래퍼 구현, 데이터 수집, DB 저장]
[코드 작성 및 실행]
"40개 카페 정보를 수집했습니다. admin/review에서 확인하세요."
```

```
User: "웹에서 카페 정보 가져와서 DB에 넣어줘"

AI: "네, 웹에서 정보를 가져오는 방법은 여러 가지가 있습니다.
먼저 어떤 웹사이트를 사용할지 결정해야 하는데요,
Naver 블로그, Tistory, 아니면 일본의 Tabelog 같은 곳도 있습니다.
각각의 장단점을 설명드리면..."
```
