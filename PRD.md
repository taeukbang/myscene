# PRD: Photo-first Place Module

## 1. 배경 및 문제 정의

### 1.1 배경

* 3040 여성 해외여행자는 여행지 선택 시 **사진 결과물**을 강하게 중시함
* 인스타그램/유튜브 등에서 카페·장소 콘텐츠는 넘치지만,

  * 실제 여행 중 다시 찾기 어렵고
  * 현재 위치/시간/동선과 맞지 않아 실행으로 이어지지 않음

### 1.2 문제 정의 (Problem Statement)

> 여행 중 "사진 찍기 좋은 장소"를 찾고 싶지만,
> 지금 이 순간에 **실제로 가서 성공할 수 있는 선택**을 빠르게 내리기 어렵다.

이로 인해:

* 저장만 하고 방문하지 않음
* 구글맵/네이버맵으로 다시 흩어짐
* 선택 피로 누적

---

## 2. 제품 목표 (Goals)

### 2.1 사용자 목표

* 장소 선택 시 고민 최소화
* "여기 가면 이 사진을 찍을 수 있다"는 확신 제공
* 여행 중 하루 1회 이상 자연스럽게 사용

### 2.2 비즈니스 목표

* Travel Concierge의 핵심 진입 모듈 확보
* 여행 중 DAU 증가
* 향후 일정/커머스/컨시어지 기능 확장 기반 마련

---

## 3. 성공 지표 (Success Metrics)

### North Star Metric

* **여행 중 사용자 1명당 Photo-first Place Module 평균 진입 횟수**

### 보조 지표

* 사진 카드 → 상세 진입률
* 상세 → 일정 저장 전환율
* 여행 1회당 재방문 일수

---

## 4. 타겟 사용자

### Primary Target

* 30–49세 한국 여성
* 해외 자유여행 (혼행/친구/부부)
* 여행 중 사진 촬영 및 SNS 공유 니즈 보유

### 사용 맥락

* 여행 중 빈 시간 (오후/저녁)
* 다음 장소를 고민하는 순간

---

## 5. 핵심 컨셉

### 제품 한 문장 정의

> "이 장소에 가면, 이 구도의 이 정도 사진을 얻는다."

### 핵심 차별점

* 장소 정보보다 **사진 결과물 중심**
* 리뷰보다 **성공/실패 조건 명시**
* 지도보다 **상황 기반 추천**

---

## 6. 사용자 플로우 (User Flow)

1. 앱 진입
2. Photo-first Place Module 자동 노출
3. 상황 카드 확인 (사진 중심)
4. 사진 상세 (촬영 가이드 확인)
5. 일정에 저장 또는 바로 길찾기
6. 여행 중 반복 진입

---

## 7. 기능 요구사항 (Functional Requirements)

### 7.1 메인 화면 – 상황 기반 사진 카드 (카페 + 뷰 스팟)

**기능 설명**

* 사용자 위치, 시간대를 기반으로 사진 카드 1–2개 노출

**구성 요소**

* 대표 사진 (Best Shot)
* 간단한 태그 (예: #오후빛 #혼자여행)
* CTA: "이 사진 찍으러 가기"

**제약**

* 리스트/무한 스크롤 금지
* 한 화면 내 최대 2개 카드

---

### 7.2 사진 상세 화면 – 촬영 가이드

**기능 설명**

* 해당 장소에서 촬영 가능한 사진의 조건을 구체적으로 안내

**구성 요소**

* 사진 전체 화면
* 촬영 가이드

  * 위치 (예: 창가 2번 테이블)
  * 시간대 (예: 오후 2–4시)
  * 구도 (정면/측면/인물 포함 여부)
* 실패 조건

  * 웨이팅 많음
  * 역광 시간대

---

### 7.3 지도 화면 (서브)

**기능 설명**

* 사진 기반 핀 노출

**구성 요소**

* 사진 썸네일 핀
* 클릭 시 사진 상세로 이동

---

### 7.4 일정 연동

**기능 설명**

* 사용자의 여행 일정에 장소를 저장

**세부 기능**

* 자동 이동 시간 계산
* 추천 방문 시간대 표시
* 일정 불가 시 경고 메시지

---

### 7.5 재방문 트리거

**기능 설명**

* 맥락 기반 알림 제공

**알림 조건 예시**

* 시간대 변경 (오전 → 오후)
* 위치 이동 (다른 지역 진입)
* 일정 완료 후 빈 시간 발생

---

## 8. 데이터 구조 (초안)

### Place

* place_id
* 이름
* 위치 (lat/lng)
* 카테고리 (카페 / 뷰 스팟)
* 지역 태그 (시부야, 아사쿠사 등)

### Photo

* photo_id
* place_id
* 이미지 URL (원본 저장 X, 캐시 또는 리사이즈)
* 출처 플랫폼 (Instagram / YouTube 등)
* 출처 계정명
* 촬영 시간대 태그
* 구도 태그 (인물/풍경/테이블 등)
* 실패 조건 태그
* 반응 지표 (좋아요 수, 저장 수 등)

### Photo Source Pipeline (자동 수집)

* 한국어 기반 여행/카페 키워드 세트 정의

  * 예: "도쿄 카페", "도쿄 감성 카페", "도쿄 뷰 카페" 등
* 공개 SNS 피드 중 한국어 콘텐츠 탐색
* 반응 지표(좋아요/저장/댓글)를 기준으로 1차 필터링
* 이미지 내 장소명/해시태그 기반 장소 추정
* 지도 API를 통한 장소 좌표 매칭
* 수동 검수 단계를 거쳐 서비스 반영

## 9. MVP 범위

> MVP 기준 도시: **도쿄 (Tokyo)**
> MVP 기준 장소 유형: **카페 + 뷰 스팟**

> MVP 기준 장소 유형: **카페 + 뷰 스팟**

### 포함

* 도시 1곳
* 장소 50–100개
* 사진 카드 / 상세 / 일정 저장
* 수동 + 룰 기반 추천

### 제외

* 리뷰 시스템
* 좋아요/댓글
* 자동 개인화 고도화

## 10. 향후 확장 방향

* 장소 유형 확장 (카페 → 뷰 스팟 → 레스토랑/바)
* Travel Concierge 추천 엔진과 통합
* UGC 사진 업로드
* 커머스 연계 (투어/체험)

## 11. 리스크 및 대응

### 리스크 1: 저작권 이슈

* SNS 이미지 무단 사용으로 인한 분쟁 가능성

**대응 전략**

* 모든 사진을 "촬영 예시 이미지"로 명시
* 출처 플랫폼 및 계정명 명확히 표기
* 이미지 다운로드/저장 기능 미제공
* 원본 고해상도 미저장
* 삭제 요청 시 24시간 내 즉시 대응 프로세스 구축

### 리스크 2: 자동 수집 데이터의 정확도

* 장소 오매칭 가능성

**대응 전략**

* 자동 수집 후 반드시 수동 검수 단계 포함
* MVP 단계에서는 반자동 운영 전제
